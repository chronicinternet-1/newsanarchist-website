What they're not telling you: # Web Developer's Query String Ban Reveals Privacy Blindspot in Mainstream Tech Discourse A web developer has publicly eliminated query strings from their site architecture—a technical decision that exposes how casually the internet industry treats user privacy in ways mainstream tech coverage routinely overlooks. The developer, documented on Hacker News and detailed in a post on susam.net titled "I've banned query strings," made a deliberate choice to remove query parameters from URLs entirely. Query strings are the portion of web addresses that appear after a question mark, containing data like search terms, user IDs, tracking parameters, and session information.

The Take
Jordan Calloway · Government Secrets & FOIA

# THE TAKE: "Banned Query Strings" Is Security Theater Susam Samanta's crusade against query parameters is purist gatekeeping dressed as principle. Yes, URLs should be "clean"—but banning them entirely? That's cargo-cult web development. Query strings aren't the problem. Unvalidated input is. Susam's blog doesn't acknowledge that REST best practices *require* parameters for filtering, pagination, sorting. You can't build functional APIs without them. Google, AWS, Stripe—they all use query strings at scale. The real receipts: his position conflates aesthetic preference with security doctrine. No CVE database shows "query string" as root cause. Proper escaping and validation do the work. His approach abandons proven patterns for theoretical purity. This is what happens when someone solves a personal problem and declares it universal law. It isn't contrarian; it's just impractical. **Grade: F for implementation reality.**

What the Documents Show

While tech publications occasionally mention query strings in passing, they rarely examine why a developer would eliminate them entirely or what that elimination reveals about surveillance embedded in web infrastructure. Query strings exist in plain text within URLs. This means they're stored in browser history, server logs, cached pages, and transmitted through various systems with minimal protection. When a user performs a search, makes a selection, or navigates through a site using query parameters, that activity creates a persistent digital trail accessible to anyone with access to these logs or history files. The mainstream tech narrative frames query strings as simply convenient—a way to pass information through URLs.

🔎 Mainstream angle: The corporate press either ignored this story entirely or buried it in a 3-sentence brief. The framing, when it appeared at all, focused on process rather than impact.

Follow the Money

It rarely emphasizes that this convenience comes at a cost: the normalization of exposed user behavior data at the architectural level of the web itself. The developer's decision to ban query strings represents a different approach: using POST requests, hidden form fields, or other methods that don't embed data into URLs where it persists indefinitely. This is technically more complex and requires deliberate security-conscious design choices. Yet mainstream coverage of web development rarely positions privacy-first architecture as a notable trend or ethical imperative. Instead, the industry standard remains the convenience-first approach, where query strings remain the path of least resistance. What makes this noteworthy isn't the technical implementation—it's what the decision reflects about the gap between privacy rhetoric and privacy practice across the technology industry.

What Else We Know

Companies frequently claim they prioritize user privacy while building systems that rely on exposed, persistent data trails by default. A developer choosing to eliminate query strings is choosing to swim against that current, suggesting that privacy-conscious design remains exceptional enough to warrant explicit public declaration. The broader implication extends beyond individual websites. The normalization of query string usage represents how surveillance mechanisms become embedded not through dramatic, centralized decisions but through thousands of small architectural choices that accumulate into a system where user behavior is perpetually logged and exposed. Mainstream tech discourse treats this as neutral infrastructure rather than as a design choice with privacy consequences. For ordinary internet users, the difference is significant.
